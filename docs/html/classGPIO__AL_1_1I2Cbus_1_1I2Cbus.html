Detailed Description

Class to handle I2C bus communication.

The GPIO Pins on the Raspberry Pi are GPIO 2 for I2C data and GPIO 3 for I2C clock for hardware I2C and freely selectable for software I2C (bit banging); on the Raspberry Pi Pico, the GPIO Pins for I2C communication are also freely selectable under software I2C, they are restricted to GPIO 1 for data and GPIO 2 for clock, GPIO 4 for data and GPIO 5 for clock, GPIO 6 for data and GPIO 7 for clock, GPIO 9 for data and GPIO 10 for clock, GPIO 11 for data and GOPI 12 for clock, GPIO 14 for data and GOPI 15 for clock, GPIO 16 for data and GPIO 17 for clock, GPIO 19 for data and GPIO 20 for clock, GPIO 21 for data and GPIO 22 for clock, GPIO 24 for data and GPIO 25 for clock, GPIO 26 for data and GPIO 27 for clock, or GPIO 31 for data and GPIO 32 for clock.

Since many targets can be connected on an I2C bus, one I2Cbus object must be able to handle them all. Therefore, I2Cbus objects are created one per I2C bus, which is uniquely defined by the sda and scl Pins - NOT one such object per target on that bus. Every I2C I/O operation therefore needs to be given the I2C address of the target this communication is meant for.

On the Raspberries running under Linux-like operating systems, it is mandatory that the user be part of the group i2c to be able to use the I2C bus. This can be accomplished by issuing the command (in a terminal window)

sudo usermod -a -G i2c <user name>

and then logging out and back in again. Otherwise, access to the I2C device requires elevated privileges (sudo), and the practice of using those when not strictly necessary is strongly discouraged.

Also, the pigpiod daemon needs to run on Raspberry Pis running under an operating system in order to use GPIO Pins. Either use

sudo pigpiod

whenever you want to use it or enable the daemon at boot time using

sudo systemctl enable pigpiod

to have the GPIO daemon start every time the Raspberry Pi OS boots.

It is worth noting that the defaults for the operating mode are different between the Raspberry Pi 3 and the Raspberry Pi Pico. This is because either the Broadcomm BCM2835 chip, which the Raspberry Pi 3 uses for hardware I2C, or the standard Raspberry Pi driver smbus is broken and does not (reliably) support clock stretching when requested by a target. This was found through measurements and also confirmed online at https://www.advamation.com/knowhow/raspberrypi/rpi-i2c-bug.html. The software mode supports clock stretching properly on the Raspberry Pi 3. Naturally, the software-generated I2C clock on a non-real-time OS is not very consistent, but targets will tolerate a non-consistent clock better, albeit not always completely, than a broken clock-stretch mechanism when they need it. Therefore, the default operating mode on a Raspberry Pi 3 is software, but since the Raspberry Pi Pico works just fine with hardware I2C, the default operating mode there is hardware to free the CPU from the task of generating I2C signals. This software still allows the caller to select hardware mode also on the Raspberry Pi 3, but the user is strongly advised to make sure that no target on the I2C bus requires clock stretching in such cases. Moreover, the user is much more likely to run into error conditions on a Raspberry Pi 3 I2C bus than on any other system, and it is a very good idea to write "robust" code that checks error conditions continuously and deals with them appropriately when the I2C bus and clock stretching has to be used on a Raspberry Pi 3, even in software mode.

Default operating mode on the Pico is hardware so as not to overburden the CPU with tasks the hardware can do.

The BCM2835 chip or the standard Raspberry Pi driver smbus is broken and the default operating mode for it is therefore set to software.
